### src/api/trackEvent.js:18 — NEEDS-REVIEW
**Commit:** `fdec65ba` feat: add privacy-respecting anonymous event tracking (#1321)
**Change summary:** New module. Writes a random UUID to `localStorage` under
`veyrnox-device-id` and POSTs `{device_id, event, metadata}` rows to a Supabase
`events` table. Gated on `!supabase || DEMO || isDeniabilityOrDemoActive()`.

**Assessment — the gate itself is correct.** `getOrCreateDeviceId()` is called
*after* the gate, so a decoy/demo-only session never creates the key and never
emits. This is exactly the shape the K-2 fix (PR #1262) established for
`referralApi.js`, and `src/api/__tests__/trackEvent.test.js` pins it — including
two explicit `expect(localStorage.getItem('veyrnox-device-id')).toBeNull()`
assertions for the DEMO and deniability paths. I3 holds.

**The gap: `veyrnox-device-id` is not in any panic-wipe residue list.**
`src/wallet-core/panic.js` erases `localStorage` by an explicit key list
(`DENIABILITY_RESIDUE_KEYS` / `METADATA_RESIDUE_KEYS` / `LOCAL_RESIDUE_KEYS`)
plus a runtime prefix sweep whose only prefix is `veyrnox-snapshots-`. There is
no `localStorage.clear()`. Repo-wide grep finds `veyrnox-device-id` in exactly
two places — `trackEvent.js` and its test — so panic wipe leaves it in place.
Two consequences:

1. **Local residue / deniability tell.** A key named `veyrnox-device-id`
   surviving on a wiped device proves a Veyrnox install existed there. This is
   the same class of artifact `DENIABILITY_RESIDUE_KEYS` already wipes for
   weaker tells — e.g. `veyrnox-autolock-timeout`, `veyrnox-message-signing-enabled`,
   and the M-7 live-prices pref, all listed precisely because their presence
   betrays the install.
2. **Server-side linkage across the wipe (I5).** Because the ID survives, the
   *next* real session's `session_start` / `wallet_created` rows are inserted
   under the **same** `device_id` as the pre-wipe rows. Panic wipe's stated goal
   is to "RAPIDLY and IRREVERSIBLY destroy the LOCAL device copy"; that promise
   is local by construction, but the backend is untrusted by design, and the
   events table now carries a join key that spans the wipe. `events.sql` has an
   index on `(device_id, created_at desc)` for exactly that retention query.

**Not claimed:** no key material, address, balance, or wallet count is
transmitted, and no wallet content distinguishes real from decoy. This is a
residue + linkability finding, not a key-exposure one.

**Suggested remediation (not applied — this task is read-only):** add
`'veyrnox-device-id'` to `DENIABILITY_RESIDUE_KEYS` in `src/wallet-core/panic.js`,
which fixes both the erase path and `inspectKeyMaterial().clean` at once (per
the F-04 note in that file).

### src/pages/SendCrypto.jsx:1145 — NEEDS-REVIEW
**Commit:** `fdec65ba` (#1321)
**Change summary:** On a successful send, emits
`trackEvent(EVENT.SEND_COMPLETED, { currency: selectedWallet?.currency })`.
`src/pages/ReceiveCrypto.jsx:45` similarly emits `RECEIVE_VIEWED` with
`{ asset }` for known assets.

**Assessment:** No address, amount, or txid leaves the device — the metadata is
a chain symbol only. But combined with the server-side `created_at` timestamp
and the persistent `device_id`, the backend accumulates a per-device timeline of
*which chain* was transacted on and *when*. Against an untrusted backend (I5),
a send timestamp plus a chain name is a correlation vector for narrowing which
on-chain transaction belongs to a device — the same reasoning that gates live
price feeds behind an explicit opt-in (M-7). Worth an owner decision on whether
`send_completed` should carry the currency at all, or be dropped to a bare event
name (or removed, since the flow is already covered by the opt-in local audit
log at `recordAudit("send_completed")` one line above).

Note this is inherent to the design, not a regression: it is a new capability
being weighed, not a control that was removed.

### android/app/build.gradle:127, android/gradle.properties:24 — SAFE (net hardening)
**Commits:** `4150a961` (#1310), `a1dc9bf6` (#1313), `7bf314bd` (#1314),
`c6009b24` (#1319)
**Change summary:** Adds a `doFirst` gate on `bundleRelease`/`assembleRelease`
that hard-fails if `RELEASE_CERT_SHA256` is blank or not 32 colon-separated hex
pairs, and if release signing did not actually resolve to an existing keystore.
`gradle.properties` no longer ships the Android **debug** keystore fingerprint as
the default value. `versionCode` 4 → 6.

**Assessment:** Clear improvement. Previously a release build could inherit the
committed debug fingerprint and ship a RASP `detectTamper` that validated
against the wrong cert; now a misconfigured build cannot be produced at all
(fail-closed, I4). The `sc.storeFile == null` check closes the narrower case
where `keystore.properties` merely *exists* — which is what selects
`signingConfigs.release` — but carries unreadable keys, which historically could
fall through to debug signing. `keystore.properties` and `*.jks` are confirmed
git-ignored; the new `keystore.properties.template` contains placeholders only.
versionCode 6 respects the "1–5 consumed" constraint.

Minor, non-security: the template uses `storeFile`/`storePassword`/`keyAlias`/
`keyPassword`, which build.gradle reads only as the *fallback* spelling — the
documented primary names are `KEYSTORE_PATH`/`KEYSTORE_PASSWORD`/`KEY_ALIAS`/
`KEY_PASSWORD`. Both resolve, so nothing breaks; the naming is just inconsistent
with the comment directly above it.

### supabase/events.sql — SAFE
**Commit:** `fdec65ba` (#1321)
**Change summary:** New table with RLS enabled and an `anon`-role
insert-only policy; no SELECT/UPDATE/DELETE grant.

**Assessment:** Correct least-privilege shape — the app cannot read its own
events back, so a compromised client cannot enumerate the table. The file
itself documents the residual honestly ("the anon key is public, so anyone with
it can spam INSERTs… tamper-resistant, not abuse-proof"), which is the right
disclosure rather than an overclaim. `scripts/verify-events.mjs` actively
asserts the anon client *cannot* read back.
