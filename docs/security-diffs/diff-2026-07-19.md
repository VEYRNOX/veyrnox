# Addendum — toolchain & repo-hygiene findings (same day)

The scan above covers the commit diff. This addendum covers three findings that surfaced
from investigating the repository state itself rather than the diff. None is a defect in
shipped wallet code; none touches key material, signing, or the I1–I5 invariants at
runtime. Two are dev-environment integrity issues and one is a near-miss on secret
handling. Recorded here because a security log that only reads diffs will never catch
this class.

## A-1 — Every Claude Code hook in the repo was a silent no-op — NEEDS-REVIEW

**Status:** FIXED — PR #1218, merged `05885844`. Verified on `main`: 1 hook entry remains
(the intended `SessionStart` cloud-sandbox `sh -c`), zero references to `hook-handler` or
`auto-memory-hook`, and `permissions`/`env` unchanged.

`.claude/settings.json` configured 16 hook entries across `PreToolUse`, `PostToolUse`,
`UserPromptSubmit`, `SessionStart`, `SessionEnd`, `Stop`, `PreCompact`,
`SubagentStart`/`SubagentStop`, and `Notification`. All of them invoked
`.claude/helpers/hook-handler.cjs` or `auto-memory-hook.mjs`.

**Neither script exists** — not in the repo, not at the `%USERPROFILE%` fallback. There
is no `helpers/` directory at all. Every one of those hooks has therefore been doing
nothing.

**Why this belongs in a security log:** the security-relevant property is not what the
hooks did, it is that anyone reading `settings.json` would reasonably conclude that
pre-tool-use gating, prompt routing, and session-boundary handling were in force. They
were not. A hook that appears configured but never executes is precisely the "fake
security" failure mode the project's own hard rules prohibit (I4 — fail honest). No
evidence was found that these hooks were relied on for a specific control; the finding is
the false impression, not a demonstrated bypass.

**Assessment:** NEEDS-REVIEW rather than REGRESSION — nothing regressed, the hooks appear
never to have worked in this configuration. The remediation removes them so the config
states the truth. If the tooling is wanted back, the helpers must be restored *and* the
quoting fixed, with a missing helper failing loudly instead of silently.

## A-2 — Malformed hook commands drop into interactive `cmd.exe` and write files into the repo — NEEDS-REVIEW

**Status:** FIXED at source — PR #1218, merged `05885844`. Root cause of a recurring
artifact seen since at least 2026-06-27. Note that hook configuration is read at session
start, so sessions already running at merge time continue to produce artifacts until they
restart; PR #1211's pre-commit hook keeps those out of commits in the meantime.

Each broken hook nests double quotes inside an already-quoted `cmd /c "..."` argument,
which cmd's `/c` parser does not handle. Combined with the missing helpers (A-1), the net
effect is that **cmd.exe starts interactively and reads the hook's JSON payload from stdin
as a command line.**

JSON escapes inner quotes as `\"`. cmd does not honour backslash escaping — each quote
character simply flips its quote state. At odd parity, a `>` anywhere in the payload
(commonly from a JavaScript arrow function `=>` quoted in a prompt or tool result) falls
outside quotes and is parsed as a **redirection**, creating a zero-byte file named after
the following token, in the repository root.

Reproduced 2026-07-19:

```
{"hook_event_name":"UserPromptSubmit","prompt":"assert toContain(\"x\")
 and \"open then (el) => el.classList.contains('line-through') done"}
→ creates: el.classList.contains('line-through')   (0 bytes)
```

Byte-identical to an artifact observed in the working tree. The cmd banner and prompt echo
this produces also matches the hook output visible in every session transcript.

**Impact.** These artifacts have reached `main` at least four times — `392f9d30`
(2026-06-27, 6 files), `ec459a6e` (2026-07-02), and `f518ba57`/`7ffe7d37` (2026-07-18).
The mechanism is *arbitrary file creation in the repo root driven by prompt and tool
content*. In practice it has only ever produced empty files with junk names, and a
redirect target is a single token so an attacker-influenced path is constrained — but the
write is real, it is driven by untrusted-ish content, and it is not something a wallet
repository should tolerate. Rated NEEDS-REVIEW, not REGRESSION: the behaviour is
long-standing, not newly introduced, and no evidence was found of it overwriting a
tracked file (a redirect to an existing tracked path would truncate it — this has not been
observed, but is not structurally prevented).

**Note on an earlier hypothesis.** An initial pass in this session attributed the
artifacts to "unquoted JS reaching bash." That was wrong about the vector: the payload is
JSON and the shell is `cmd.exe`, invoked by the hook layer, not by any Bash tool call.
The corrected mechanism above is the reproduced one. Recording the correction because the
wrong version circulated in this session's commit messages before the root cause was
found.

## A-3 — Android upload signing keystore was untracked but not ignored — NEEDS-REVIEW (closed)

**Status:** FIXED — PR #1215, merged `86d78cdf`.

`veyrnox-upload-new.jks` was sitting in the working tree, untracked but **not** matched by
any `.gitignore` rule. The existing keystore rules were exact filenames
(`veyrnox-release.jks`, `keystore.jks`) plus `*.keystore`, which does not match a `.jks`
extension. The file was one `git add -A` from being committed.

**No leak occurred.** `git log --all -- "*.jks"` confirms no keystore has ever been
committed, and the file was neither staged nor tracked when found.

**Assessment:** genuine near-miss on a live-mainnet wallet. Had it been committed and
pushed, remediation would mean a Play Console upload-key reset, with an exposure window
running from push to detection. Closed by replacing the exact-filename rules with
`*.jks` / `*.p12` / `*.jceks` globs. Verified no currently-tracked file matches the new
globs, so nothing silently dropped out of version control. This is exactly the class of
near-miss A-2 makes more likely, since broad `git add -A` is the shared failure path.

## A-4 — Required status checks on `main` are bypassable by admin, and were bypassed today — NEEDS-REVIEW (governance)

**Status:** open. Not a misconfiguration — a deliberate setting whose practical effect is
worth stating explicitly.

`main` has **no classic branch protection** (`/branches/main/protection` → 404), but it is
covered by an active ruleset, `Veyrnox Code Review` (id `17946638`,
`enforcement=active`):

| Rule | Setting |
|---|---|
| `required_status_checks` | `verify`, `mainnet-flag-gate` |
| `pull_request` | 1 approving review, dismiss stale reviews on push |
| `code_scanning` | CodeQL, security alerts ≥ high, alerts ≥ errors |
| `deletion`, `non_fast_forward` | blocked |

The ruleset has one bypass actor:

```
actor_type=RepositoryRole   actor_id=5 (admin)   bypass_mode=pull_request
```

**Observed today.** PR #1218 merged at 14:16:00Z with `verify` and `web-e2e-tests` still
`pending` on the PR head. Auto-merge had been enabled on that PR and would have waited —
auto-merge honours required checks — so the merge was an admin-role merge that bypassed
the requirement. (`mergedBy` is not diagnostic here: it reports the same account whether
auto-merge fires or a human merges, because auto-merge was armed under that account.) The
`verify` run visible on merge commit `05885844` — started 14:16:03, `cancelled` 14:16:06 —
is the post-merge run on `main`, superseded when `39f8962c` landed on top. It is **not**
evidence that `verify` passed on the PR head.

**Why it matters.** `verify` is the job that runs typecheck, `check:strays`, and
`check-cert-pin-manager-safety`. It is the specific gate that catches the class of failure
that cost time earlier today: a typecheck break sat on `main` (see the `Subscription.test.jsx`
`TS2556`/`TS2345` errors that blocked PR #1211 until `main` was fixed). Bypassing `verify`
is how a red `main` arises in the first place.

**Assessment.** NEEDS-REVIEW as governance, not as a defect. The gating is correctly
configured; admin bypass is a deliberate escape hatch, and CLAUDE.md documents admin
override as routine for this repo — the `pull_request` rule requires 1 approving review,
which a single-collaborator repo cannot satisfy without it. So the bypass is not
misuse; it is the only way this repo merges anything.

For #1218 specifically the residual risk was negligible: 15 deletions in a single JSON
file, no source or test code, with `lint-and-build`, `test-status`, `CodeQL`, and
`mainnet-flag-gate` all green before the merge.

**Suggested posture (owner decision, not actioned):** treat admin bypass as reserved for
provably inert diffs, and let `verify` complete on anything touching source, tests, or
build config. Alternatively, drop `required_approving_review_count` to 0 so the
status-check requirement can be satisfied without needing bypass at all — that would make
`verify` genuinely binding rather than advisory, which is the stronger fix.

**Correction recorded.** An earlier statement in this session asserted that `verify` was
"evidently not a required check," reasoning from the classic branch-protection 404 alone.
That was wrong: rulesets are a separate enforcement mechanism and had to be queried
separately. `verify` is and was required. Noted here because the incorrect claim was
stated before the ruleset was checked.
