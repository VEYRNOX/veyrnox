  the residue class that produced finding #1094 is closed here.
- The unlock block is wrapped in `catch { }` and never blocks unlock, so a decoy session
  produces no observable timing or behavioural tell (deniability-symmetric).

Two observations, neither a violation:
1. **Defence-in-depth gap (LOW).** The I3 guard exists only at the API layer. The
   codebase's established belt-and-braces pattern (PRs #783, #858, #921) normally pairs a
   primitive-layer guard with a caller-site guard. The `WalletProvider` call sites have no
   independent check. Not exploitable as written — the chokepoint holds — but it is one
   refactor away from being the only thing standing between a decoy session and Supabase.
2. **Functional, not security (LOW).** In the unlock block `clearPendingReferral()` is
   called *before* the `try`, so if the first unlock after install is a decoy/demo session,
   `redeemCode` throws on the I3 guard, the `catch` swallows it, and the pending referral
   code is silently consumed and lost. Correct from a deniability standpoint; a referral
   attribution bug from a product standpoint.
