## Flagged changes

### src/api/referralApi.js — SAFE
**Commit:** e86d5d6a fix(api): rate-limit and harden all Supabase RPC endpoints (#1334)
**Change summary:** Every referral call migrates from direct table access (`.from().insert/upsert/select`)
to `SECURITY DEFINER` Postgres RPCs (`generate_referral_code`, `register_referral_code`,
`increment_referral`, `record_attribution`, `get_referral_paid_count`, `get_referral_earnings`).
Each write/read now passes a device id from the new `@/lib/deviceId` helper.
**Assessment:** Security-positive. Adds no client-side attack surface and removes direct
anon table writes. Critically, every exported function keeps its `isDeniabilityOrDemoActive()`
guard as the first statement, and `getOrCreateDeviceId()` is called **only after** that gate
returns — so I3 (deniability → zero backend calls) is preserved and no device id is minted or
sent in a decoy/demo session. The device-id linkage is new data on referral RPCs but is (a)
gated by deniability, (b) the same identifier already sent by `trackEvent`, and (c) used for
server-side rate limiting / increment dedup, not for key material. No I1/I2 regression.
Cosmetic only: `increment_referral` still uses the legacy `ref_code` arg name alongside
`p_device_id` while sibling RPCs use `p_code` — inconsistent, not a defect.

### src/lib/deviceId.js — SAFE (new file)
**Commit:** e86d5d6a (#1334)
**Change summary:** New module factoring the previously-duplicated device-id logic into one
`getOrCreateDeviceId()` with an in-memory cache. Same crypto-random UUID generation and
`localStorage` persistence as before.
**Assessment:** Pure refactor + memoisation, byte-for-byte equivalent generation logic. The
module itself makes no network call and no deniability decision — every caller gates before
invoking it. Fails closed (returns `null`) on any exception, and all callers treat `null` as
a no-op / 500. No egress on its own.

### sql/api-security-hardening.sql — SAFE (server-side, not shipped)
**Commit:** e86d5d6a (#1334)
**Change summary:** Backend migration: `track_event` (event-name allowlist, ≤64-char event,
≤4 KiB metadata, 60/device/hour rate limit, silent drop over cap), `increment_referral`
(mandatory `device_id`, `referral_increments` dedup table with no anon RLS → max 1 increment

it, so I5 posture is unchanged and improved server-side.
