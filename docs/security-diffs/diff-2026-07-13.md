### src/pages/CryptoSigning.jsx — NEEDS-REVIEW
**Commit:** `53f36610d` feat(signing): wire /crypto-signing to real wallet behind off-by-default toggle (#852)
**Change summary:** Large rewrite — removes the ephemeral key generation/import path (where the page held a random `ethers.Wallet` in a `useRef`) and replaces it with `withPrivateKey(0, async (pk) => { ... })` from `useWallet()`, scoping the private key inside the signing callback exactly as `WalletConnectProvider` does. Message signing is additionally gated behind `useMessageSigningEnabled()` (off by default) and `isUnlocked`.

**Assessment:**

This is the highest-sensitivity change in today's batch because it connects `/crypto-signing` to the **real session wallet** rather than an ephemeral throwaway key.

*Positive security changes:*
- The old path held an `ethers.Wallet` object (including `.privateKey`) in a `useRef` for the lifetime of the component — a persistent in-memory exposure. The new path scopes the private key to `withPrivateKey`'s callback lifetime only. Architecturally better for I1.
- Off-by-default toggle (`useMessageSigningEnabled`) limits the attack surface.
- `isUnlocked` guard is correct.
- RASP gate (`raspGuardAllowsSigning`) is preserved from the old implementation.
- `makeCopy` now calls `copyPlain` unconditionally, which is correct: the only values copied are `address` (public) and `signature` (public). The old path had a `{ sensitive }` routing that could copy mnemonics and private keys — those code paths are gone entirely.

*Open questions for review:*
1. **Deniability session** — `withPrivateKey(0, ...)` signs with the *active session's* wallet. In a decoy/hidden session this signs with the decoy key, which is correct. The comment in the code says so. However, there is no explicit `isDeniabilitySessionActive()` guard on this page. The send flow has belt-and-suspenders I3 guards at the provider layer; this page bypasses the provider layer entirely (direct `ethers.Wallet.signMessage`). Verify that the decoy/hidden wallet is the one returned by `useWallet()` in those sessions — if so the deniability property is inherited, not explicitly enforced here.
2. **`withPrivateKey` index 0** — hardcodes account index 0. This matches the active EVM account but could sign with the wrong key if a multi-account user expects a different account. Not a security regression versus the old ephemeral path, but worth noting.
3. **No on-chain txid** — the toggle and real-wallet wiring are noted as off-by-default; the change is `BUILT` at most. Do not mark as `verified`.

**Rating: NEEDS-REVIEW** — specifically the deniability session inheritance. The change is almost certainly safe (decoy session unlocks a decoy wallet, which is what `useWallet()` returns), but this should be explicitly verified in code or a test. No hard I1–I5 violation confirmed, but the I3 path is implicit rather than explicit here.
