### src/pages/CryptoSigning.jsx — SAFE ✅ (resolved 2026-07-13, commit `41c81ea89`)
**Commit:** `53f36610d` feat(signing): wire /crypto-signing to real wallet behind off-by-default toggle (#852)
**Change summary:** Large rewrite — removes the ephemeral key generation/import path (where the page held a random `ethers.Wallet` in a `useRef`) and replaces it with `withPrivateKey(0, async (pk) => { ... })` from `useWallet()`, scoping the private key inside the signing callback exactly as `WalletConnectProvider` does. Message signing is additionally gated behind `useMessageSigningEnabled()` (off by default) and `isUnlocked`.

**Assessment:**

This is the highest-sensitivity change in today's batch because it connects `/crypto-signing` to the **real session wallet** rather than an ephemeral throwaway key.

*Positive security changes:*
- The old path held an `ethers.Wallet` object (including `.privateKey`) in a `useRef` for the lifetime of the component — a persistent in-memory exposure. The new path scopes the private key to `withPrivateKey`'s callback lifetime only. Architecturally better for I1.
- Off-by-default toggle (`useMessageSigningEnabled`) limits the attack surface.
- `isUnlocked` guard is correct.
- RASP gate (`raspGuardAllowsSigning`) is preserved from the old implementation.
- `makeCopy` now calls `copyPlain` unconditionally, which is correct: the only values copied are `address` (public) and `signature` (public). The old path had a `{ sensitive }` routing that could copy mnemonics and private keys — those code paths are gone entirely.

*Deniability session gap — CLOSED (commit `41c81ea89`):*

The original code contained a comment claiming "decoy/hidden sessions sign with their own active wallet — that is correct, not special-cased here." This reasoning was wrong. A signature is a verifiable cryptographic commitment to a specific identity; producing one in a deniability session undermines plausible deniability regardless of which key is used, because the signature can be presented as evidence that a wallet exists and was active.

**Fix (`fix(signing): explicit I3 deniability guard on CryptoSigning page`):**
- Imports `isDeniabilitySessionActive` from `@/wallet-core/deniabilitySession.js`
- **Render guard (C4):** deniability session renders a locked "not available in this session" panel — address, message input, and Sign button are never shown. Copy is neutral and does not disclose the session type.
- **Sign-time guard:** `signMsg()` checks `isDeniabilitySessionActive()` before calling `withPrivateKey`; fail-closed (I4) — sets an error message and returns without signing.
- **Tests:** `CryptoSigning.i3.test.jsx` (5/5) covering baseline, render guard, and sign-time defense-in-depth. All CryptoSigning specs 29/29 passing.

**Rating: SAFE** — all open questions resolved. BUILT / unit-tested, INTERNAL.
