> **Decision note тАФ M2c/d OS-enforced key binding:**
>
> Today (M2b, native.js): vault ciphertext is stored in the platform hardware-backed store (iOS Keychain / Android Keystore) with ThisDeviceOnly + passcode-gated accessibility; unlock is gated by an APP-LAYER biometric prompt (authenticate in JS, then read + decrypt the blob). Vault crypto is the unchanged Argon2id+AES-GCM format, byte-identical to web.
> Gap (M2c/d): the gate is app-layer, NOT an OS-enforced ACL bound to the stored item тАФ kSecAttrAccessControl(biometryCurrentSet) on iOS / setUserAuthenticationRequired (+ setIsStrongBoxBacked where available) on Android. App-layer means in-context code that skips the JS check could read the blob; OS-ACL means the hardware itself refuses to release/decrypt without a fresh biometric and invalidates if the enrolled biometric set changes.
>
> **M2c iOS scaffold landed (PRs #690/#695, 2026-07-07):** the Swift plugin (`VeyrnoxEnclavePlugin`, `EnclaveKeyService`, `EnclaveError`) + JS bridge (`veyrnoxEnclave.js`) implement `kSecAttrTokenIDSecureEnclave` + `SecAccessControlCreateWithFlags([.privateKeyUsage, .biometryCurrentSet])`. The M2c up-migration also now logs a failed write via `logM2cMigrationFailure` (`native.js`) instead of silently swallowing `VAULT_WRITE_VERIFY_FAILED` (KEK-audit **M-3**). Both fixes are BUILT / unit-tested, INTERNAL. F-2 status: **BUILT, not verified** тАФ device-test + independent audit both still required; the remaining activation blocker is the **M-6** design decision (iOS-F5 `NSString hB64` bridge copy тАФ architectural) тАФ see the "2026-07-07/08 INTERNAL KEK stack audit" table below.
> Best-of-breed design: layer OS-ACL binding ON TOP OF the existing password-derived key as a stronger gate, never a replacement. The password path MUST remain the independent recovery route тАФ if the OS-bound key were the only gate, a biometric reset or device migration would invalidate the ACL and permanently destroy the vault (fund loss). Mirrors the existing passkey/biometric escape-hatch rule: password is always THE secret, hardware is a layer. isSecureHardwareAvailable() must report OS-ACL availability truthfully per-device (StrongBox absent on most Android; SE only on real iPhones); the UI must never claim OS-enforced protection on a device that only has app-layer тАФ degrade to the software vault and say so.
> Build constraint: requires a thin custom native plugin (current Capacitor plugins do not expose per-item biometric ACL binding) тАФ Swift (iOS SE/Keychain) + Kotlin (Android Keystore/StrongBox). Not buildable or verifiable in the JS/web environment.
> Verification gates (what "verified" requires тАФ none satisfiable in JS):
> 1. Build native app with the plugin; install on a REAL device with the hardware (physical iPhone w/ SE; Pixel 3+ / recent Samsung w/ StrongBox). Emulators/simulators have no secure hardware and cannot verify this.
> 2. Functional: enroll biometric, lock, confirm the OS blocks decrypt without a fresh biometric; confirm a biometric-set change invalidates per biometryCurrentSet.
> 3. Adversarial (the real test): attempt to read the stored item WITHOUT satisfying the biometric (e.g. a debug build skipping the JS gate) and confirm the OS still refuses. This distinguishes OS-ACL from app-layer; a JS test cannot exercise it.
> 4. Confirm the password fallback still recovers the vault after an ACL invalidation (no fund-loss footgun).
> 5. Independent audit sign-off (key-at-rest is core crypto; expands audit scope per native.js).
> JS-seam tests (interface contract, capability-gating fallback, no-plaintext-caching) are worth writing WHEN the plugin exists, with the native layer mocked тАФ they verify the code's USE of the hardware, not the hardware guarantee itself.

## 2026-07-13 Android RASP improvements тАФ Magisk-Hide-bypass detection vectors (DEVICE-VERIFIED, INTERNAL, 2026-07-14)

> тЬЕ DEVICE-VERIFIED (INTERNAL, 2026-07-14) тАФ re-tested on the SM-N981B (Samsung Galaxy Note 20 5G, Magisk v30.7, Android debug build) at 00:20:12. Three new Magisk-Hide-bypass detection vectors added to `RaspIntegrityPlugin.kt` to address the `rooted:false` gap observed during the 2026-07-12 Magisk v30.7 session (expected at the time тАФ Magisk Hide masks file paths the original checks query). The existing three checks are unchanged.

**New checks:**
- **`checkProcNetUnix()`** тАФ reads `/proc/net/unix` for known root-framework IPC socket names (`@magisk_`, `magiskd`, `@ksu_`, `zygisk`, `@lspd`, `apatchd`). These are kernel-level IPC entries, not filesystem objects under Magisk's mount-namespace control тАФ analogous to the iOS `checkJailbreakPathsCstat` (C `stat()` bypassing `NSFileManager`'s sandbox filter).
- **`checkSuFromRuntime()`** тАФ executes `which su` via `Runtime.getRuntime().exec()`. On stock Android, `which su` produces no output. On rooted devices where Magisk Hide does not cover this app, the path is returned. Analogous to the iOS `checkFork()` behavioral test.
- **`checkDangerousProps()`** тАФ reads `ro.boot.verifiedbootstate` and `ro.boot.flash.locked` via `android.os.SystemProperties` reflection (in-process, no `exec`, no SELinux denial). An unlocked bootloader reports `verifiedbootstate=orange` or `flash.locked=0`; these values come from the bootloader and are baked into the kernel command line, so Magisk Hide does not intercept them. **SELinux root cause (found and fixed in commit `f46abecba`):** the initial implementation used `Runtime.exec("getprop ...")`, which is denied by SELinux policy for `untrusted_app` on Android 10+ and silently returns empty тАФ causing `rooted:false` in early testing. Replaced with `readSystemPropReflect()` using `android.os.SystemProperties` via reflection, which is an in-process call not subject to that denial.

**Extended detection lists:**
- `checkRootBinaries` and `checkMagiskPaths` extended with KernelSU (`/data/adb/ksud`, `/proc/ksud`), Apatch (`/data/adb/apatch`), LSPosed (`/data/adb/lspd`), and newer Magisk artifacts (`/data/adb/magisk_db`, `/dev/.magisk.unblock`, `/data/adb/modules`).
- `checkXposed` extended with LSPosed Manager (`org.lsposed.manager`) and KernelFlasher (`me.weishu.kernelflasher`).
- `checkProcMapsForHook` extended with `zygisk` and `lspd` markers.
