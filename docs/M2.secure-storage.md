# M2 — Native Secure Storage + Biometrics (DETAILED SPEC)

> The security-critical mobile phase: move key storage from the web IndexedDB
> vault to hardware-backed storage (iOS Secure Enclave/Keychain, Android
> Keystore/StrongBox) with biometric unlock. This is BOTH a security upgrade and
> an Apple-review requirement. Build on a Mac (after M1 runs on simulators).
>
> RULES: strict pre-merge PR review (this touches key handling — no shortcuts).
> Additive to web (web vault path unchanged). No crypto-algorithm changes — same
> BIP-39/32/44 + Argon2id + AES-GCM; only WHERE the key/vault lives changes.
> Testnet only; mainnet stays gated. M2 goes in the audit scope.

---

## Problem M2 solves

Self-review (docs/SECURITY_SELFREVIEW_FINDINGS.md) confirmed the core crypto is
sound, with ONE cross-cutting limitation: **on web, secrets live transiently as
immutable JS strings that can't be zeroized, and the vault sits in IndexedDB
protected only by the password.** On mobile we can do better: hardware-backed
key storage + biometric gating, so the password is not the sole factor and key
material is protected by the Secure Enclave / Keystore.

---

## THE KEY ARCHITECTURAL DECISION: what goes in the hardware?

Two valid designs — pick deliberately. RECOMMENDATION: **Design B** (wrap-key in
hardware), as it's the standard, most portable, and keeps your existing audited
vault format.

### Design A — store the whole vault/seed in the keystore
- The encrypted vault (or even the seed) is written directly into Keychain /
  Android Keystore as a secured item, with biometric access control.
- Simpler conceptually; but item-size limits, less control over the crypto, and
  ties more of the design to platform specifics.

### Design B — hardware-wrapped key (RECOMMENDED)
- Keep the existing Argon2id+AES-GCM vault FORMAT (already audited-friendly).
- Generate a hardware-backed key INSIDE the Secure Enclave / StrongBox that
  never leaves hardware; use it to wrap (encrypt) the vault-encryption key (or
  to gate access). Biometric unlock authorises the hardware key's use.
- The seed/vault is decrypted only after biometric + hardware unlock; key
  material is reconstructed transiently in app memory only for signing.
- Pros: reuses your vetted vault, password OR biometric as factors, standard
  pattern, portable across both OSes via one abstraction.

> Whichever is chosen, INVARIANT: the seed/private key is NEVER persisted in
> webview storage (IndexedDB/localStorage) on native, and never written to logs.

## Architecture: one interface, per-platform implementations

Keep `vaultStore` (and key-access) as a STABLE INTERFACE; swap implementation by
platform at runtime:

```
keyStore interface (conceptual):
  hasVault(): bool
  saveVault(blob): void           // ciphertext only
  loadVault(): blob               // ciphertext only
  unlock(): Promise<secret|grant> // triggers biometric on native
  lock(): void                    // clear in-memory material
  isSecureHardwareAvailable(): bool

implementations:
  - web    → existing IndexedDB + Argon2id vault (UNCHANGED)
  - native → Capacitor plugin bridging iOS Keychain/Secure Enclave +
             Android Keystore/StrongBox, with biometric gating
```

Platform detection via Capacitor (`Capacitor.isNativePlatform()` / getPlatform()).
The wallet-core crypto (mnemonic/derivation/signing) does NOT change — it just
receives the secret from whichever store unlocked it.

## Plugin choice (evaluate, don't assume)
- Prefer a VETTED, maintained Capacitor plugin for secure storage + biometrics
  rather than hand-rolling native code. Candidates to evaluate (verify current
  maintenance/security at build time — do not assume): capacitor-secure-storage
  plugins, biometric-auth plugins (e.g. community/official). 
- Criteria: actively maintained, uses Keychain+Secure Enclave (iOS) and
  Keystore+StrongBox (Android) properly, supports biometric access control,
  reasonable audit history, permissive licence.
- If no plugin meets the bar, a thin custom Capacitor plugin (Swift + Kotlin) is
  acceptable but EXPANDS the audit scope — note it.

## iOS specifics
- **Keychain** item with `kSecAttrAccessControl` requiring biometric
  (`biometryCurrentSet` / `userPresence`), `kSecAttrAccessibleWhenUnlocked...
  ThisDeviceOnly` (no iCloud sync of secrets).
- **Secure Enclave** for the hardware key (Design B): generate a non-exportable
  EC key in the Enclave; use it to wrap/unwrap the vault key.
- **Face ID / Touch ID** via LocalAuthentication; handle no-biometry-enrolled
  and lockout fallbacks (passcode fallback policy is a deliberate decision).

## Android specifics
- **Android Keystore**, prefer **StrongBox** (`setIsStrongBoxBacked(true)`) where
  available, fall back gracefully if not.
- Key with `setUserAuthenticationRequired(true)` + biometric; choose auth
  validity duration deliberately (per-use is strongest).
- **BiometricPrompt** for unlock; handle no-enrolled-biometric + lockout.
- Ensure secrets are not in `allowBackup`/auto-backup.

## Verification gates (review these before merge + in the audit)
- [ ] Seed/private key NEVER in webview storage (IndexedDB/localStorage) on
      native — inspect device storage + grep.
- [ ] Hardware-backed key is non-exportable; never leaves Enclave/Keystore.
- [ ] Biometric required to unlock / authorise a transaction; lockout + no-
      enrollment fallbacks behave safely (deliberate passcode policy).
- [ ] Secret reconstructed in memory only for the signing op; cleared on lock /
      background / idle. (Improves the web JS-lifetime limitation.)
- [ ] No secret in logs/analytics/crash reports (grep + runtime check).
- [ ] Web vault path UNCHANGED; existing 58 tests still green; check:rng green.
- [ ] Crypto core (mnemonic/derivation/signing) byte-identical — M2 only changes
      storage/unlock, not algorithms.
- [ ] iOS: ThisDeviceOnly accessibility (no iCloud secret sync); Android: no
      auto-backup of secrets.
- [ ] Plugin (or custom native code) added to the independent-audit scope.
